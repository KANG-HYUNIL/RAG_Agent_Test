#### 전제

- 결과 저장 방식, 파싱 항목, 실행 방식은 1차~3차 실험과 동일
- 3차 실험에서 일부 run은 API rate limit 영향으로 시간 지표가 오염되었으므로, 4차-1은 상위 후보를 다시 검증하는 clean confirmation 단계로 정의
- 4차-1의 목적은 새로운 하이퍼파라미터 탐색이 아니라, 상위 후보들의 정확도와 category별 성능을 안정적으로 재확인하는 것
- hit@k 분석, 오답 수작업 분류도 진행 시도

#### 목적

- 3차 상위 후보의 정확도를 clean rerun으로 재검증
- `Law` / `Criminal Law` category별 성능 차이를 확인
- retrieval trace를 기반으로 hit@k를 계산할 수 있는 분석용 산출물 확보
- 상위 후보에 대해 오류 유형을 분류할 수 있는 기반 마련

#### 공통적으로 확인된 내용

| 항목 | 공통 결론 |
| --- | --- |
| category 적합성 | 전반적으로 높음. top-1 category는 대부분 올바름 |
| 핵심 병목 | category mismatch보다 같은 분야 안의 near-miss retrieval이 더 큼 |
| 반복 패턴 1 | `옳은 / 옳지 않은 / 아닌 것은 / 틀린 것은` polarity 혼동 반복 |
| 반복 패턴 2 | `(○)/(×)를 바르게 연결한 것은` 같은 템플릿 유사성 과적합 반복 |
| 반복 패턴 3 | 표면적으로 매우 비슷하지만 정답 근거로는 부족한 문항 retrieval 반복 |
| 반복 패턴 4 | `question_plus_choices`가 성능 향상에는 도움되나, 일부 문항에서는 선택지 표면어에 끌림 |
| threshold 효과 | threshold 계열 실험도 실제 filtering 효과는 매우 약함 |

#### 실험 간 차이점

| 비교 항목 | k5_t035 | k5_t045 | topk_k5 | k7_t030 |
| --- | --- | --- | --- | --- |
| 전체 성능 | 최고 | `035`보다 소폭 낮음 | 더 낮음 | 가장 낮음 |
| Law | 준수 | 가장 높음 | 낮음 | 낮음 |
| Criminal Law | 가장 나음 | 낮음 | `035`와 동일 | `045`와 동일하게 낮음 |
| filtering | 거의 없음 | 약간 더 있음 | 없음 | 없음 |
| noise 유입 | 보통 | 보통 | 큼 | 가장 큼 |
| 해석 | 가장 균형적 | 더 보수적이나 실익 제한적 | baseline 확인용 | retrieval 수 증가의 한계 확인용 |

#### 핵심 문제 패턴 정리

| 문제 패턴 | 의미 | 이번 실험에서의 해석 |
| --- | --- | --- |
| polarity 혼동 | 정답 방향이 반대인 유사 문항이 검색됨 | 가장 반복적이고 직접적인 오류 원인 |
| 템플릿 과적합 | 형식이 비슷한 문제를 과하게 끌어옴 | 의미보다 문제 형식 유사성이 강하게 작용 |
| near-miss retrieval | 같은 분야의 매우 유사한 문제를 가져오지만 정답 근거로는 부족 | category는 맞지만 답 판단에는 불충분 |
| 옵션 끌림 | 선택지 단어가 retrieval에 과하게 영향 | 핵심 개념보다 선택지 표면 표현에 끌리는 사례 존재 |
| retrieval 수 증가 부작용 | 더 많이 가져올수록 정보가 좋아지는 것이 아님 | `k=7`은 recall보다 noise 증가 효과가 더 큼 |

#### 종합 비교 결론

- 4개 실험 모두 category 자체는 대체로 잘 맞춤
- 그러나 정답 근거를 정밀하게 고르는 데에는 공통적인 한계가 있음
- 핵심 문제는 category mismatch가 아니라
  1. polarity 혼동  
  2. 템플릿 유사성 과적합  
  3. near-miss retrieval  
  4. 선택지 표면어 끌림  
  에 가까움
- `k5_t035`가 현재까지는 가장 균형적인 후보로 판단됨
- `k5_t045`는 더 보수적이지만 실질적 개선은 제한적임
- `topk_k5`는 baseline으로서 의미는 있으나 noise 제거가 되지 않음
- `k7_t030`은 retrieval 수를 늘리면 오히려 precision과 context 품질이 더 나빠질 수 있음을 보여줌
