### 260409_제5차 실험 진행 순서

#### 전제

- 결과 저장 방식, 파싱 항목, 실행 방식은 1차~4차 실험과 동일
- 4차 실험 결과, 현재 병목은 category mismatch보다
  - polarity 혼동
  - 템플릿 유사성 과적합
  - near-miss retrieval
  - 선택지 표면어 끌림
  에 더 가까운 것으로 판단함
- `k=7` 확장은 성능 개선보다 noise 증가 효과가 더 크게 나타났으므로, 5차에서는 `final_k=5`를 유지함
- 5차의 핵심은 프롬프트 수정이 아니라, retrieval 자체를 더 정확하게 만드는 것에 둠



#### BM25 Hybrid란?

- 현재 retrieval은 dense retrieval 중심임
- dense retrieval은 의미적으로 비슷한 문항을 잘 찾지만,
  - `옳은 것은` ↔ `옳지 않은 것은`
  - `아닌 것은`
  - `틀린 것은`
  같은 짧고 중요한 표면 표현 차이를 충분히 반영하지 못하는 경우가 있음
- BM25는 query와 문서 간의 실제 단어 일치를 강하게 보는 lexical retrieval 방식임
- 따라서 hybrid retrieval은
  - dense retrieval의 의미 유사성
  - BM25의 exact match / 법률 용어 / polarity 표현 일치
  를 함께 활용하는 방식으로 이해함

> **왜 공백 split이 아닌 Kiwi를 쓰는가?**
> 한국어는 교착어로, 조사(격조사·보조사)가 어절에 결합된다. 공백 split을 하면 `출석의`, `출석을`, `출석에서`가 서로 다른 BM25 토큰으로 처리되어 같은 개념임에도 매칭이 실패한다. Kiwi로 형태소 분석하면 세 표현 모두 `출석(NNG)`으로 분리되어 동일 토큰으로 인식된다.

##### Kiwi 형태소 분석기 설정

Kiwi 인스턴스는 모듈 레벨 singleton으로 관리한다 (`get_kiwi()`). 프로세스당 1회만 초기화되어 초기화 비용(~1초)이 실험 전체에서 1회만 발생한다.

**Windows 비ASCII 경로 문제 대응**: Kiwi의 C++ 백엔드는 경로에 한국어 등 비ASCII 문자가 포함되면 모델 파일을 열지 못한다. 이를 `GetShortPathNameW` Win32 API로 8.3 단축 경로로 변환하여 우회한다.

```python
# _get_kiwi_model_path() 내부 로직 (요약)
ctypes.windll.kernel32.GetShortPathNameW(model_dir, buf, 1024)
# 예: "C:\...\kiwipiepy_model" → "C:\KIWIPI~1"
Kiwi(model_path=short_path)
```

**법률 복합명사 사용자 사전 등록**: `kiwi.add_user_word(word, "NNG", 0.0)`으로 44개 법률 복합명사를 NNG(일반명사)로 등록하여 분절을 방지한다.

          공동불법행위, 표현대리, 위법수집증거, 재량행위, 취소소송 ...

##### BM25 문서 텍스트 구조화

단순 `question + A/B/C/D` 결합이 아니라, **structured sparse text** 형식으로 구성한다.

```
"{polarity_tag} {question_noun_tokens} {choice_noun_tokens} {cat_token}"
```

- `polarity_tag`: `"부정방향"` / `"긍정방향"` / `"중립"` — polarity를 BM25 term으로 삽입하여 방향성 alignment 보강
- `question_noun_tokens`: 질문의 Kiwi NNG/NNP 토큰
- `choice_noun_tokens`: A/B/C/D 선택지의 Kiwi NNG/NNP 토큰
- `cat_token`: `"형사법"` / `"민사법"` — category 구분 신호

> polarity_tag 삽입 이유: 부정 방향 질문(`옳지 않은 것은?`)이 전체의 63%를 차지하므로, 쿼리와 문서가 같은 polarity 방향일 때 BM25 점수가 올라가 polarity mismatch를 줄일 수 있음.

##### BM25 인덱스 빌드 시점

`Retriever.add_documents()` 완료 후 `strategy.post_add_documents()` hook이 호출된다. `HybridRetrievalStrategy`는 이 hook을 override하여 **Phase 1(인덱싱)에서 BM25 인덱스를 eager 빌드**한다. 벤치마크 루프 안에서 지연 없이 동작한다.

```
benchmark.py
  └── retriever.add_documents(chunks, embeddings)  ← FAISS 인덱스 구축
        └── strategy.post_add_documents(documents) ← BM25 인덱스 구축 (1회)
              └── BM25Okapi(tokenized_docs)
```

- 빌드 소요: 학습 데이터 2073건 기준 약 **3~10초** (1회, Phase 1에 흡수)
- query당 추가 연산: Kiwi 토크나이징 ~1ms + BM25 스코어링 ~5ms ≈ **약 6ms 추가** (LLM 호출 1~2초 대비 무시 가능)

##### Dense + BM25 Hybrid Search 로직 (RRF)

```
1. Dense 검색: FAISS IndexFlatIP (L2-normalized cosine) → top-10 후보
2. Sparse 검색: BM25Okapi.get_scores(query_tokens) → 점수 상위 top-10 후보
3. RRF 합산: score(d) = Σ 1 / (60 + rank_i)
   - 각 ranking에서 문서 d의 순위 rank_i(1-based)에 대해 합산
   - k=60은 표준 RRF 상수 (Cormack et al., 2009)
4. 중복 제거: 동일 doc_idx는 dense rank + sparse rank의 RRF 점수가 합산됨
5. 최종 정렬 후 top_k=5 반환
```

**RRF 수식**:

```
RRF_score(d) = 1/(60 + rank_dense) + 1/(60 + rank_sparse)
                ↑ dense에만 있으면  ↑ sparse에만 있으면
               둘 중 하나만 합산    둘 중 하나만 합산
```

두 시스템에 모두 등장한 문서는 양쪽 점수가 합산되어 자연스럽게 가중치를 받는다. `dense_top_k=10`, `sparse_top_k=10`이므로 최대 20개 후보에서 `final_k=5`를 추출한다.

##### Query 텍스트 BM25 전달 경로

```
benchmark.py
  └── query_text = build_query_text(method, question, choices)
  └── retriever.search(query_vector, query_text=query_text)
        └── tokenize_korean(query_text) → query_tokens
        └── strategy.set_query_tokens(query_tokens)
        └── strategy.search(index, documents, query_np)
              ├── FAISS.search(query_np, dense_top_k)
              ├── BM25Okapi.get_scores(query_tokens)
              └── RRF → top_k 반환
```

`Retriever.search()`에 `query_text: str | None = None` 파라미터를 추가하여 BM25 query를 전달한다. 기존 top_k / score_threshold / mmr 전략은 `set_query_tokens()`가 no-op이므로 하위 호환성이 완전히 유지된다.

#### Query Reformulation이란?

- query reformulation은 원본 질문의 뜻을 바꾸는 것이 아니라, retrieval이 더 잘 되도록 query를 retrieval 친화적으로 다시 구성하는 것임
- 현재 실험에서는 `question_plus_choices`가 이미 효과를 보였으나,
  - polarity 정보가 약하고
  - 선택지 표면어에 과하게 끌리는 사례가 있었음
- 따라서 5차에서는 query를 더 구조적으로 재구성하여
  - 핵심 법개념
  - 문제의 정답 방향성
  - 선택지 정보
  가 retrieval에 더 잘 반영되도록 시도함

##### polarity_aware_qpc 구현

`question_plus_choices`에 polarity 방향 태그를 prefix로 추가하는 방식이다. 질문 의미와 선택지 내용은 그대로 유지하되, 문제의 방향성을 명시적으로 추가한다.

**출력 형식**:
```
"[{polarity_tag}] {question} A) ... B) ... C) ... D) ..."
```

**polarity_tag 값**:

| 값 | 의미 | 해당 표현 예시 |
| --- | --- | --- |
| `부정방향` | 틀린/아닌/해당하지 않는 것 선택 | 옳지 않은 것은, 아닌 것은, 틀린, 잘못된, 허용되지 않는 |
| `긍정방향` | 맞는/해당하는/옳은 것 선택 | 옳은 것은, 해당하는, 인정되는, 성립하는, 타당한 |
| `중립` | OX 연결형 또는 판정 불가 | 옳은 것(○)과 옳지 않은 것(×)을 바르게 연결한 것은 |

**detect_polarity() 로직 (우선순위)**:

```
1. OX 연결형 선감지
   re.search(r"옳은.{0,10}옳지\s*않은", question)
   → 매칭 시 즉시 "중립" 반환 (NEG/POS 패턴이 모두 포함되어 있어 잘못 분류되는 것 방지)

2. 부정 패턴 (말미 40자 우선 → 전체 검사)
   "옳지\s*않은", "해당하지\s*않는", "아닌\s*것", "틀린",
   "잘못\s*된", "적절하지\s*않은", "인정되지\s*않는",
   "성립하지\s*않는", "허용되지\s*않는", "없는\s*것",
   "타당하지\s*않은", "맞지\s*않는", "올바르지\s*않은"
   → 매칭 시 "부정방향" 반환

3. 긍정 패턴 (말미 40자 우선 → 전체 검사)
   "옳은\s*것", "해당하는\s*것", "맞는\s*것", "인정되는",
   "성립하는", "허용되는", "있는\s*것", "타당한",
   "적절한", "올바른"
   → 매칭 시 "긍정방향" 반환

4. 나머지 → "중립"
```

**말미 40자 우선 검사 이유**: 한국어 문장은 서술어가 문미에 위치한다. `옳은 것은?`/`옳지 않은 것은?` 등 polarity 표현이 질문 끝에 집중되므로, 말미 40자를 먼저 검사하면 오탐(질문 본문 속의 수식어 `없는 경우`, `아닌 때` 등을 잘못 포착하는 경우)을 줄일 수 있다. 말미에 없으면 전체 검색으로 fallback한다.

**polarity 분포 (train 2073건 기준)**:

| polarity | 건수 | 비율 |
| --- | --- | --- |
| 부정방향 | ~1470 | ~63% |
| 긍정방향 | ~560 | ~24% |
| 중립 | ~43 | ~13% |

부정 방향이 63%로 지배적이다. polarity 태그를 query에 추가하면 dense embedding이 `[부정방향]` prefix를 공유하는 문서를 더 높은 유사도로 매칭할 수 있다.

**변환 예시**:

| 원본 query (question_plus_choices) | polarity_aware_qpc |
| --- | --- |
| `피고인 출석의 예외에 대한 설명으로 옳지 않은 것은? A) ... B) ...` | `[부정방향] 피고인 출석의 예외에 대한 설명으로 옳지 않은 것은? A) ... B) ...` |
| `불법행위로 인한 손해배상청구권에 관한 설명으로 옳은 것은? A) ...` | `[긍정방향] 불법행위로 인한 손해배상청구권에 관한 설명으로 옳은 것은? A) ...` |

##### core_focus_query 구현

선택지를 완전히 제거하고, 질문에서 핵심 법률 명사만 추출한다. 선택지 표면어가 dense embedding에 noise를 유발하는 경우를 방지하는 것이 목표다.

**출력 형식**:
```
"[{polarity_tag}] {핵심명사1} {핵심명사2} ..."
```

**extract_core_tokens() 로직**:

```
1. Kiwi.analyze(question) → 형태소 분석 (singleton 재사용, 초기화 비용 없음)
2. 보존 품사: NNG, NNP, SH, SL, SN (NNB 완전 제거)
3. 길이 < 2 토큰 제거
4. _LEGAL_STOPWORDS 적용 (설명, 경우, 것, 수, 판례 등)
5. 중복 제거 (출현 순서 유지)
```

> **NNB(의존명사)를 완전 제거하는 이유**: `것`, `수`, `바` 등의 의존명사는 `tokenize_korean()`(BM25용)에서는 stopwords로 걸러지지만, `extract_core_tokens()`에서는 품사 단계에서 원천 차단한다. `core_focus_query`의 목표는 핵심 법률 개념어만 남기는 것이므로, 의존명사를 별도 필터 없이 품사 레벨에서 제거하는 것이 더 안전하다.

**fallback 동작**: Kiwi 분석이 실패하거나 핵심 토큰이 0개인 경우, `"[{polarity_tag}] {question}"` (원문)으로 자동 fallback한다. 실험 중단 없이 안전하게 처리된다.

**변환 예시**:

| 원본 질문 | core_focus_query |
| --- | --- |
| `피고인 출석의 예외에 대한 설명으로 옳지 않은 것은?` | `[부정방향] 피고인 출석 예외` |
| `불법행위로 인한 손해배상청구권의 소멸시효에 관한 설명으로 옳은 것은?` | `[긍정방향] 불법행위 손해배상청구권 소멸시효` |
| `민법상 혼인의 무효사유가 아닌 것은?` | `[부정방향] 민법 혼인 무효 사유` |

선택지가 제거되어 query가 매우 짧아지지만, 핵심 법률 개념어 밀도가 높아져 의미 공간에서 정확한 문서 영역을 지향할 것으로 기대한다.

##### query 구성 방식 비교

| 방법 | 선택지 포함 | polarity 태그 | 형태소 분석 | 예상 효과 |
| --- | --- | --- | --- | --- |
| `question_only` | X | X | X | 기준 (너무 짧음) |
| `question_plus_choices` | O | X | X | 4차 best |
| `polarity_aware_qpc` | O | O (regex) | X | polarity mismatch 완화 |
| `core_focus_query` | X | O (regex) | O (Kiwi) | 선택지 noise 제거 + polarity 보강 |

##### BM25 query와의 연관

`core_focus_query`는 dense embedding query(임베딩 벡터 생성용)이지만, hybrid 전략에서 BM25 query로는 **원본 `question_plus_choices` 텍스트**를 `tokenize_korean()`으로 분석한 토큰이 사용된다. 즉, BM25 레이어와 dense 레이어의 query 텍스트는 독립적으로 관리된다.

```
dense query (임베딩):  "[긍정방향] 불법행위 손해배상청구권 소멸시효"
BM25 query (토큰):    ['불법행위', '손해배상청구권', '소멸시효', ...]
→ 두 레이어가 각각 최적화된 형태로 검색에 참여
```

#### 종합 비교 결론

- 5차 실험의 세 변형(`hybrid_qpc`, `dense__polarity_qpc`, `hybrid__polarity_qpc`) 모두 기준선 `baseline_dense_qpc`를 넘지 못함
- 따라서 현재 시점의 최선 후보는 여전히 `baseline_dense_qpc`로 보는 것이 적절함
- 다만 이번 실험은 “새 방향이 전부 무의미했다”기보다, 아래를 확인해준 진단 실험으로 해석하는 것이 맞음
  1. BM25 hybrid 방향 자체는 가능성이 있으나, 현재 구현은 표면 유사성 과적합이 큼  
  2. polarity 문제는 실제로 존재하지만, 현재 prefix 태그 방식은 해결책이 아님  
  3. same-category retrieval이 높아도 정답이 오르지 않는 만큼, 병목은 category가 아니라 세부 쟁점 정밀성 부족에 있음
- 즉, 이번 5차 실험의 핵심 결론은  
  **“문제의식은 맞았지만, 현재 hybrid 구현과 polarity 주입 방식은 채택하기 어렵다”**  
  로 정리하는 것이 가장 적절함
