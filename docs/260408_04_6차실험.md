### 260408_제6차 실험 진행 순서

#### 전제

- 결과 저장 방식, 파싱 항목, 실행 방식은 1차~5차 실험과 동일
- 5차 실험 결과, 현재 최강 기준선은 여전히 `stage5__baseline__dense_qpc`임
- 5차 실험에서 확인된 핵심 문제는
  - polarity 혼동
  - 템플릿 유사성 과적합
  - near-miss retrieval
  - same-category retrieval이 높아도 정답 근거 정밀성이 부족한 점
  에 가까움
- 따라서 6차의 핵심은 **새 retriever를 또 바꾸는 것**이 아니라,  
  **현재 최강 dense retrieval 결과를 더 정밀하게 다시 고르는 것**에 둠
- `k=7` 단순 확대는 오히려 noise를 키웠으므로, 6차에서는 `retrieve_k`와 `final_k`를 분리하여 검토함

#### 왜 6차는 retrieval replacement가 아니라 retrieval correction인가?

- 5차에서 `hybrid_qpc`는 baseline을 넘지 못했고, `dense__polarity_qpc`는 오히려 성능을 떨어뜨렸음
- 즉,
  - BM25는 **주력 candidate generator**로 쓰기에는 표면 유사성 과적합이 컸고
  - polarity 태그 prefix는 **dense query 자체를 흐릴 가능성**이 있었음
- 하지만 동시에,
  - BM25가 전혀 무의미한 것은 아니고
  - polarity 문제도 실제 병목으로 확인됨
- 따라서 6차에서는
  - BM25를 **보조 rerank 신호**
  - polarity를 **후단 penalty/filter 신호**
  로 낮춰 써서, baseline dense retrieval을 정밀 보정하는 방향으로 감

#### 6차 핵심 설계 개념

##### 1. Dense baseline 유지
- candidate generation은 `score_threshold + question_plus_choices` 기준선을 그대로 사용
- 즉, retrieval backbone은 `stage5__baseline__dense_qpc`를 계승함

##### 2. BM25 auxiliary rerank
- BM25는 더 이상 dense와 동등한 2번째 retriever로 쓰지 않음
- dense가 먼저 뽑은 후보들에 대해서만 BM25 score를 계산하여 보조 점수로 사용
- 목적:
  - 법률명 / 제도명 / 기관명 / exact 표현 일치 신호 보강
  - full hybrid에서 발생한 candidate contamination 완화

##### 3. Polarity-aware penalty
- `[부정방향]`, `[긍정방향]` prefix를 query에 직접 넣지 않음
- 대신 retrieval 후단에서
  - query polarity
  - candidate polarity
  를 비교하여 opposite-polarity 후보에 penalty를 부여
- 목적:
  - `옳은 것은` ↔ `옳지 않은 것은`
  - `아닌 것은`
  - `틀린 것은`
  같은 반대 방향 retrieval 완화

##### 4. Template penalty
- `(○)/(×)`, `바르게 연결한 것은`, `설명으로 옳은 것은` 같은 형식 유사성이 과하게 작동하는 경우를 완화하기 위해
  - 템플릿은 유사하지만
  - 핵심 법개념 overlap이 낮은 문항
  에 penalty를 부여
- 목적:
  - OX/연결형 문제에서 “같은 시험 형식”만 닮은 문항이 top에 올라오는 현상 억제

##### 5. Retrieve more → re-select
- baseline dense retrieval은 `retrieve_k`를 다소 넉넉히 가져오고
- rerank / penalty 적용 후 `final_k`만 prompt에 전달
- 목적:
  - recall은 유지하되
  - final context의 precision을 높임

#### 목적

- 5차 최고 후보인 `baseline_dense_qpc`를 기반으로, 후단 정밀도 보정을 통해 정확도 개선 가능성을 확인
- BM25를 candidate generator가 아닌 auxiliary rerank 신호로 사용할 때의 효과 확인
- polarity 혼동 완화가 실제 정확도 개선으로 이어지는지 확인
- 템플릿 과적합 완화가 OX/연결형 문항에 도움되는지 확인
- `retrieve_k`와 `final_k` 분리 시, 더 많이 찾고 더 적게 넣는 구조가 유효한지 검증

#### 6차 retrieval 품질 검토 항목

| 항목 | 확인 목적 |
| --- | --- |
| hit@1 / hit@3 / hit@5 | retrieval 자체가 baseline보다 좋아졌는지 확인 |
| opposite-polarity 감소 여부 | polarity mismatch가 실제로 줄었는지 확인 |
| OX/연결형 오류 감소 여부 | 템플릿 과적합이 완화되었는지 확인 |
| near-miss retrieval 감소 여부 | 같은 분야 안의 빗나간 문항 비율이 줄었는지 확인 |
| BM25 auxiliary 효과 | lexical exact match가 candidate contamination 없이 보조 신호로 작동하는지 확인 |
| final_k 축소 효과 | 적은 context가 더 높은 precision으로 이어지는지 확인 |

#### 6차 결과 해석 기준

| 결과 패턴 | 해석 | 다음 단계 |
| --- | --- | --- |
| polarity penalty만 효과 있음 | polarity 문제는 맞았고, prefix 태그 방식만 잘못된 것 | polarity 후단 규칙 유지 |
| BM25 auxiliary만 효과 있음 | BM25는 retriever보다 rerank signal로 더 적합 | BM25 보조 신호 세부 조정 |
| polarity + BM25 조합이 최고 | 두 신호가 서로 보완적임 | 해당 조합을 새 기준선으로 채택 |
| final_k=3가 더 좋음 | 적게 넣을수록 noise가 줄어 precision이 올라감 | final_k 축소 방향 유지 |
| template penalty가 OX/연결형에만 효과 | 형식 과적합이 실제 병목이었음 | 템플릿 보정 규칙 유지 |
| 아무 조합도 baseline을 못 넘김 | 현재 row-to-row dense retrieval 구조 ceiling 가능성 | 이후 corpus/architecture 재검토 |

#### 6차 종합 기대 효과

- full hybrid의 candidate contamination 없이 BM25의 lexical signal만 보조적으로 활용
- polarity 문제를 query prefix가 아니라 후단 penalty로 다룸
- 템플릿 과적합을 retrieval 후단에서 직접 제어
- retrieve_k 확대와 final_k 축소를 분리하여 precision 개선 가능성 검증
- 현재 baseline을 유지하면서도, 가장 반복적인 오류 패턴에 직접 대응하는 구조 마련
